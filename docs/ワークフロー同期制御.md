# ワークフロー同期制御の詳細

## 概要

Repo Observerは単一のワークフローで構成されており、複数のジョブが順次実行されます。

## ワークフロー構成

### リポジトリ情報同期ワークフロー (`repo-full-sync.yml`)

- **実行タイミング**: 毎日 18:00 UTC（JST 3:00）
- **実行時間**: 約8-25分（リポジトリ数による）
- **処理内容**:
  1. 依存関係をビルドしてキャッシュ
  2. CSV出力を実行
  3. Issue同期を実行（順次）

## ジョブ実行フロー

```text
ワークフロー実行フロー:

┌─────────────────────────────┐
│ Job 1: build-and-prepare    │
│   - npm install             │
│   - TypeScript build        │
│   - Artifact保存            │ ← 1回だけ実行
└───────────┬─────────────────┘
            │
            ├──────────────┬──────────────┐
            ↓              ↓              ↓
┌──────────────────┐  (条件分岐)         │
│ Job 2: export-csv│                    │
│   - Artifact取得 │                    │
│   - CSV出力      │                    │
└──────┬───────────┘                    │
       │                                │
       └────────────┬───────────────────┘
                    ↓
        ┌─────────────────────┐
        │ Job 3: sync-issues  │
        │   - Artifact取得    │
        │   - Issue同期       │
        └─────────────────────┘
                    ↓
        ┌─────────────────────┐
        │ Job 4: 完了通知     │
        └─────────────────────┘
```

**ジョブの依存関係**:

- `export-csv`: `build-and-prepare`完了後に実行（スキップ可能）
- `sync-issues`: `build-and-prepare`と`export-csv`完了後に実行（スキップ可能）
- `notify-completion`: 全ジョブ完了後に実行

**メリット**:

- ビルド時間の短縮（npm install/buildを1回だけ）
- ビルド成果物の再利用（Artifact経由）
- 柔軟な実行パターン（スキップオプション）
- 効率的なリソース活用

## GitHub API レート制限について

### 制限値

- **認証済みリクエスト**: 5,000リクエスト/時間
- **GraphQL API**: 5,000ポイント/時間

### ワークフローの消費量（推定）

リポジトリ数を`N`とした場合：

| 実行パターン | API呼び出し数 | 100リポジトリの場合 |
|------------|--------------|-------------------|
| 両方実行（デフォルト） | `2N + 60`    | 約260リクエスト    |
| Issue同期のみ | `2N + 50`    | 約250リクエスト    |
| CSV出力のみ | `2N + 10`    | 約210リクエスト    |

**単一ワークフローの利点**:

- ✅ データ取得が1回で済む（重複なし）
- ✅ ジョブの順次実行により予測可能
- ✅ 最も効率的なAPI使用

## 推奨される運用方法

### 通常運用

- **自動実行に任せる**: 毎日自動実行される
- **柔軟な実行**: スキップオプションで必要な処理のみ実行可能

### 初回セットアップ時

1. **手動実行を推奨**
2. 全リポジトリを一度に登録
3. 以降は自動実行に任せる

### 実行パターン

#### 両方実行（デフォルト）

```text
自動実行 or 手動実行でパラメータはデフォルトのまま
→ CSV出力 + Issue同期
```

#### Issue同期のみ

```text
手動実行時に skip_csv: true を設定
→ Issue同期のみ実行
```

#### CSV出力のみ

```text
手動実行時に skip_issues: true を設定
→ CSV出力のみ実行
```

### トラブルシューティング

#### 問題: 特定のジョブが失敗した

**原因**: ネットワークエラー、API制限など

**対処**:

- Actions画面でログを確認
- 必要に応じて再実行
- `always()`条件により、一部失敗しても後続ジョブは実行される

#### 問題: APIレート制限に達した

**原因**: リポジトリ数が多い、または他の処理と重なった

**対処**:

1. 1時間待つ（制限がリセットされる）
2. フィルタリングオプションを使用（プライベート/アーカイブを除外）
3. リポジトリ数が非常に多い場合は実行頻度を調整

## カスタマイズ例

### 実行頻度の変更

```yaml
# 毎日実行に変更
on:
  schedule:
    - cron: "0 18 * * *"  # 毎日 18:00 UTC
```

```yaml
# 月1回実行に変更
on:
  schedule:
    - cron: "0 18 1 * *"  # 毎月1日 18:00 UTC
```

### デフォルトでCSV出力をスキップ

```yaml
skip_csv:
  description: "CSV出力をスキップ（Issue同期のみ実行）"
  type: boolean
  required: false
  default: true  # デフォルトでスキップ
```

## まとめ

単一ワークフロー+ジョブ分割のアーキテクチャにより、以下を実現：

1. **効率性** - データ取得を1回に集約
2. **一貫性** - 同じデータで全出力を生成
3. **柔軟性** - スキップオプションで部分実行可能
4. **保守性** - ロジックが1箇所に集約

この構成により、シンプルかつ効率的な運用が可能です。
